### Слайд 2 — Описание проблемы (обязательный блок)

**Какую задачу решаем**

- Люди подключают десятки сервисов (образование, госуслуги, доставка), но **теряют контроль над согласиями**: не помнят, кому что открыли.

**Симптомы**

- Не помнят список сервисов с доступом  
- Неясно, **какие категории данных** уходят третьим лицам  
- Сложно **отозвать** доступ без «охоты» по настройкам  
- Не видна **связь между системами**, которые используют одни и те же данные  

**Почему актуально**

- Чем удобнее обмен данными между сервисами, тем **менее заметна** для человека передача данных — растёт юридический и репутационный риск для организаций и stress для пользователей.

### Слайд 3 — Описание решения (обязательный блок)

**Суть Consent OS**

- **Единая панель (dashboard):** все активные согласия в одном месте.  
- **Понятные резюме** согласий простым языком (не замена юридического текста, но опора для решения).  
- **История событий** (выдано / обновлено / отозвано) — прозрачный след.  
- **Визуализация потоков данных** от субъекта данных к сервисам.  
- **Быстрое отзывание** с подтверждением и записью в историю (в MVP — логика на стороне прототипа; интеграции — мокапы).

**Три опоры**

1. **Visibility** — видеть все подключения  
2. **Understanding** — понимать категории данных и риск (эвристика, не ML)  
3. **Control** — отзыв в один шаг из одного интерфейса

### Слайд 5 — Модель решения (обязательный блок)

**Как это работает (логика продукта)**

1. **Субъект данных** управляет записями о сервисах и статусе согласия (`active` / `revoked`).  
2. Каждое действие (выдача, обновление, отзыв) попадает в **append-only журнал событий** с временной меткой.  
3. **Отзыв:** пользователь подтверждает → статус сервиса «revoked» → в журнал пишется событие `revoked` (в продукте — дальше webhook/API к поставщику услуги).  
4. **Риск:** правила по ширине набора категорий (например location + financial → higher exposure).  
5. Интеграции в кейсе смоделированы; **реальный SSO/бэкенд** — этап масштабирования.

*Мини-схема:* Пользователь → слой Consent OS (единый UX) → мокапы внешних платформ.

### Слайд 6 — User journey

1. Пользователь регистрируется в **учебной платформе**, **госпортале** и **приложении доставки**.  
2. Проходит время — в обычной среде неочевидно, что именно открыто.  
3. Открывает **Consent OS**: видит карту потоков и список активных согласий.  
4. Читает **plain-language summary** и **историю**.  
5. **Отзывает** доступ, например, у доставки — статус и журнал обновляются.

### Слайд 9 — Финансы / окупаемость идеи (дополнительно)

- **Затраты:** frontend, минимальный backend, security review при проде; интеграции по API с партнёрами.  
- **Эффект:** снижение репутационных и комплаенс-рисков, меньше обращений в поддержку «как отключить доступ».  
- **Не претендуем** на полный финансовый аудит в рамках кейса — логика **cost vs risk reduction**.

---

### Слайд 10 — Заключение

**Если Consent OS внедрить**

- Пользователь возвращает себе **контроль и прозрачность**  
- Организации получают **предсказуемый** сценарий отзыва согласий  
- Снижаются риски **злоупотреблений и «забытых» доступов**
