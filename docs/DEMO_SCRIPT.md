# Consent OS — сценарий демонстрации (~2 минуты)

Цель: закрыть критерии кейса — архитектура consent-системы, UX ясность, реалистичность внедрения (как цель + мокапы), логика демонстрации.

## Подготовка

1. Запустить локально: `npm run dev` или открыть задеплоенный URL в полноэкранном режиме.
2. Вкладка по умолчанию: **Consent dashboard**.

## Реплика по шагам

**1. Проблема → продукт (15 сек)**  
«Мы решаем потерю контроля над согласиями между образованием, госом и бытовыми приложениями. Consent OS даёт один dashboard, историю и отзыв.»

**2. Data flow map (30 сек)**  
Перейти на **Data flow map**.  
«Вот карта: от субъекта данных „You“ к трём интеграциям-мокапам. На рёбрах — категории: образование, идентификация, геолокация и оплата у доставки. После отзыва узел можно показать как отключённый — в прототипе статус „revoked“ влияет на карточку.»

**3. Consent dashboard (35 сек)**  
Вкладка **Consent dashboard**.  
«У каждого сервиса — категории данных, краткое plain-language резюме и эвристический риск: у доставки выше из‑за связки location и financial. Разворачиваем summary — это не юридический текст, а человекочитаемое объяснение для решения.»

**4. Permission history (25 сек)**  
**Permission history**.  
«Журнал append-only: выдано, обновление оплаты, и после отзыва появится событие revoked с меткой времени.»

**5. Revocation (25 сек)**  
Вернуться на **Consent dashboard**, выбрать активный сервис (например QuickDelivery), **Revoke access**.  
«Подтверждаем — статус становится revoked, в историю пишется отзыв. В продукте сюда подключили бы вызов API поставщика.»

## Чеклист оценки (самопроверка)

- [ ] Видны активные согласия и категории данных.  
- [ ] Есть визуализация потоков и история.  
- [ ] Отзыв с подтверждением и записью в истории.  
- [ ] Честно названы мокапы интеграций и демо-данные.

## Запасной план

Если интернет пропал: показать `npm run preview` после `npm run build` или записанный короткий экранный ролик с теми же шагами.
